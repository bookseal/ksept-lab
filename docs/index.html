전체 그림: 요청 하나가 흐르는 길

개별 개념에 들어가기 전에, 브라우저에서 버튼을 눌렀을 때 데이터가 어디를 거쳐 오는지부터 봅니다. 핵심은 두 개의 서버가 따로 돈다는 것 — 프론트(:5173, Vite)와 백(:5001, Flask)은 서로 다른 프로세스이자 다른 origin입니다.

flowchart LR B["Browser
(React app on :5173)"] V["Vite dev server
:5173"] F["Flask
:5001"] B -->|"GET /api/hello"| V V -->|"proxy forwards
/api/* to :5001"| F F -->|'{ message: "Hello from Flask" }'| V V --> B
CS 관점: 두 서버는 각각 별도의 TCP 리슨 소켓 (포트 5173, 5001)을 가진 독립 프로세스입니다. 브라우저 입장에서 포트가 다르면 곧 다른 origin이고, 이게 뒤에 나올 CORS와 프록시가 필요한 근본 이유입니다.

Flask

backend · python

Flask는 파이썬 마이크로 웹 프레임워크입니다. "마이크로"는 기능이 적다는 뜻이 아니라, ORM·인증 같은 걸 강제하지 않고 HTTP 요청 → 파이썬 함수 → HTTP 응답이라는 최소 골격만 제공한다는 의미입니다.

우리 코드의 @app.route("/api/hello")는 데코레이터로, "이 URL로 요청이 오면 이 함수를 실행하라"는 매핑(라우팅 테이블 등록)을 선언합니다.

그 아래에는 WSGI가 있다: Flask 앱은 사실 app(environ, start_response) 형태의 WSGI 콜러블입니다. 개발 시엔 app.run()의 내장 서버가, 운영에선 gunicorn 같은 WSGI 서버가 소켓에서 받은 raw HTTP를 environ dict으로 파싱해 이 콜러블에 넘깁니다. 즉 Flask는 "소켓을 직접 만지지 않는 요청 핸들러"라고 보면 정확합니다.

One endpoint

api · http

엔드포인트(endpoint)는 "HTTP 메서드 + 경로"의 조합 하나를 가리킵니다. 우리에겐 GET /api/hello 단 하나뿐이죠. 같은 경로라도 GET과 POST는 다른 엔드포인트입니다.

요청이 들어오면 Flask는 (메서드, 경로)로 라우팅 테이블을 조회해 맞는 함수를 호출하고, 함수의 반환값을 HTTP 응답 바디로 직렬화합니다. jsonify(...)는 파이썬 dict을 JSON 문자열로 바꾸고 Content-Type: application/json 헤더를 붙여줍니다.

sequenceDiagram participant C as Client participant R as Flask router participant H as hello() handler C->>R: GET /api/hello R->>R: match (method, path) in route table R->>H: call mapped function H-->>R: dict {message: "..."} R-->>C: 200 OK + JSON body

CORS (Cross-Origin Resource Sharing)

browser · security

브라우저에는 Same-Origin Policy가 있습니다. 스크립트가 자기 origin(스킴+호스트+포트)과 다른 origin으로 보낸 요청의 응답을 읽는 것을 기본적으로 막죠. 악성 사이트가 당신이 로그인한 은행 API를 몰래 읽는 걸 막기 위한 장치입니다.

CORS는 그 빗장을 서버가 명시적으로 풀어주는 규약입니다. 서버가 Access-Control-Allow-Origin 헤더를 응답에 실어 "이 origin은 내 응답을 읽어도 된다"고 알려줍니다. Flask에서 CORS(app) 한 줄이 이 헤더를 자동으로 붙여줍니다.

중요한 포인트: 일부 요청에서는 진짜 요청 전에 preflight(OPTIONS) 요청이 먼저 갑니다. 브라우저가 "이 메서드/헤더로 보내도 돼?"를 미리 물어보는 절차입니다.

sequenceDiagram participant JS as Browser JS (origin A) participant S as Server (origin B) Note over JS,S: Preflight (only for "non-simple" requests) JS->>S: OPTIONS /api/hello (Origin: A) S-->>JS: 204 + Access-Control-Allow-Origin: A Note over JS,S: Browser checks header, then sends real request JS->>S: GET /api/hello (Origin: A) S-->>JS: 200 + Access-Control-Allow-Origin: A Note over JS: Header present & matches -> JS may read body
그럼 우리는 왜 프록시도 쓰고 CORS도 켰나? 개발 중엔 Vite 프록시가 /api를 같은 origin처럼 보이게 만들어 CORS가 애초에 발동하지 않습니다. CORS는 프록시를 안 거치고 브라우저가 :5001을 직접 때릴 때(또는 배포 환경)를 위한 안전망입니다.
흔한 오해: CORS는 서버를 보호하는 게 아니라 브라우저가 클라이언트를 보호하는 정책입니다. 헤더가 없어도 요청 자체는 서버에 도달하고 처리됩니다 — 단지 브라우저가 JS에게 응답을 안 넘겨줄 뿐. 그래서 curl엔 CORS가 무의미합니다.

React

frontend · ui

React는 선언형 UI 라이브러리입니다. DOM을 직접 document.createElement로 명령형으로 조작하는 대신, "이 상태일 때 화면은 이렇게 생겨야 한다"를 함수로 기술하면, React가 실제 DOM을 그 모습에 맞춰줍니다.

핵심 메커니즘은 Virtual DOM diffing입니다. 상태가 바뀌면 컴포넌트 함수를 다시 실행해 새 가상 트리를 만들고, 이전 트리와 비교(diff)해 달라진 최소한의 노드만 실제 DOM에 반영(reconciliation)합니다.

flowchart TD A["state changes
(useState setter)"] --> B["re-run component fn"] B --> C["new virtual DOM tree"] C --> D{"diff vs
previous tree"} D -->|"changed nodes only"| E["patch real DOM"] D -->|"unchanged"| F["skip"]
왜 diff를 하나: 실제 DOM 조작은 레이아웃/리플로우를 유발해 비쌉니다. 메모리상의 JS 객체 트리를 비교하는 비용이 DOM을 통째로 다시 그리는 비용보다 훨씬 싸기 때문에, "가상 트리 비교 → 최소 패치" 전략이 성립합니다. 자료구조 diff 문제라고 보면 익숙할 겁니다.

Vite

frontend · tooling

Vite는 프론트엔드 빌드 도구이자 개발 서버입니다. 개발 중 가장 큰 특징은 번들링을 미루는 것: 브라우저가 지원하는 네이티브 ESM(import/export)을 그대로 활용해, 요청된 모듈만 그때그때 변환해 보냅니다. 그래서 프로젝트가 커져도 dev 서버 시작이 거의 즉각적입니다.

우리 vite.config.js의 server.proxy 설정은 dev 서버가 /api/* 요청을 받아 Flask로 대신 전달(reverse proxy)하게 합니다. 덕분에 브라우저는 모든 요청을 :5173 한 곳으로만 보내고, same-origin처럼 동작합니다.

flowchart LR subgraph Dev["vite dev (:5173)"] P["proxy rule:
/api -> :5001"] M["on-demand ESM
transform + HMR"] end BR["Browser"] -->|"/src/*.jsx"| M BR -->|"/api/*"| P P --> FL["Flask :5001"]
dev vs build는 다른 엔진: 개발 땐 esbuild(Go로 짠 초고속 트랜스파일러)로 모듈을 즉시 변환하고, 운영 빌드(vite build) 땐 Rollup으로 트리쉐이킹·번들링해 최적화된 정적 파일을 뽑습니다. "느린 번들은 배포 때 한 번만" 하자는 설계입니다.

JSX

frontend · syntax

JSX는 JS 안에 HTML처럼 생긴 마크업을 쓰게 해주는 문법 설탕(syntactic sugar)입니다. 브라우저는 JSX를 모릅니다 — 빌드 단계에서 순수 함수 호출로 변환(transpile) 됩니다.

예를 들어 <p>{message}</p>는 대략 React.createElement("p", null, message)로 바뀌고, 이 함수가 앞서 본 가상 DOM 노드(JS 객체)를 반환합니다. 즉 JSX는 결국 함수 호출이자 표현식입니다.

flowchart LR J["JSX
<p>{msg}</p>"] -->|"esbuild / babel
parses to AST"| A["AST transform"] A --> C["React.createElement('p', null, msg)"] C --> V["virtual DOM node (JS object)"]
파일 확장자가 .jsx인 이유: 트랜스파일러에게 "이 파일엔 JS 표준이 아닌 JSX 문법이 섞여 있으니 AST 파싱 규칙을 바꿔라"라고 알리는 신호입니다. 컴파일러 수업의 lexer/parser 단계를 떠올리면, JSX는 별도 문법 확장이고 .jsx는 그 파서를 켜는 플래그인 셈입니다.

Mount

react · lifecycle

마운트(mount)는 컴포넌트가 처음으로 실제 DOM에 삽입되는 순간입니다. 우리 코드의 ReactDOM.createRoot(...) .render(<App />)이 App을 #root 안에 처음 그리는 게 곧 마운트죠.

"fetch on mount"란, 화면이 처음 뜬 직후 한 번 데이터를 불러온다는 패턴입니다. 이를 위해 useEffect(fn, [])를 씁니다 — 의존성 배열이 비어 있어 마운트 시 딱 한 번만 실행됩니다.

sequenceDiagram participant R as ReactDOM.render participant App as App component participant DOM as Real DOM participant API as /api/hello R->>App: first render (mount) App->>DOM: insert markup (shows "Loading...") Note over App: after paint, effects run App->>API: useEffect([]) -> fetch once API-->>App: { message } App->>DOM: re-render with message
왜 렌더 본문이 아니라 useEffect인가: 컴포넌트 함수는 렌더할 때마다 실행되고, 순수해야 합니다(같은 입력 → 같은 출력, 부수효과 없음). 네트워크 요청 같은 side effect는 렌더 결과가 화면에 칠해진 뒤 useEffect에서 격리해 실행합니다. 빈 의존성 배열은 "이 효과는 외부 상태에 의존하지 않으니 마운트 때 한 번" 이라는 계약입니다.
참고: 현재 App.jsx의 useEffect 안에는 TODO(human)이 남아 있습니다. 바로 이 fetch-on-mount 로직을 직접 채워보는 학습 과제예요.
